ASTERRA DevOps Assignment – Technical Report

Overview

Deployed AWS infrastructure via Terraform: public ODK Central server behind an Application Load Balancer (ALB), and a private GeoJSON processing pipeline using event-driven architecture.

Key Technical Decisions

  • Infrastructure: Terraform for single-command deployment. VPC with public, private, and database subnets following AWS Well-Architected Framework.
  • Container Platform: Chose ECS Fargate over EC2 to eliminate server management and simplify operations.
  • Processing Pipeline: Used S3 → Lambda → ECS pattern for event-driven processing. Selected over EventBridge for simplicity, direct S3 integration, and low latency. Lambda validates and triggers a containerized Python/GeoPandas app.
  • Security: Enforced least-privilege IAM roles, restrictive security groups (only ALB ports 80/443), and used AWS Secrets Manager for RDS credentials. Deployment is fully automated via a single deploy.sh script.
  • CI/CD: Implemented GitHub Actions for automated versioning, Docker builds to ECR, and ECS rolling deployments. Separated test and production dependencies to optimize pipeline speed.

Primary Challenge – GDAL Dependencies

  • Problem: GDAL, required by GeoPandas, caused compilation errors across environments – local builds worked, but CI/CD pipelines failed.
  • Solution:
    • Split dependencies: requirements-test.txt (lightweight CI) vs requirements-docker.txt (full GIS)
    • Installed GDAL system libraries in Docker before Python packages
    • Set required environment variables (GDAL_CONFIG, CPLUS_INCLUDE_PATH)
  • Result: Fast CI/CD pipeline with full GIS capabilities in production.

What I Learned

Well-structured Terraform modules prevent infrastructure conflicts. Event-driven patterns elegantly decouple components. Security should be designed from the start—retrofitting IAM and network rules later is more complex. Infrastructure should scale with actual needs: start simple, then iterate based on usage.
